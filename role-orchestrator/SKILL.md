# 角色编排器 - 多代理协调

**自包含的编排，在任何用户项目中都可以在 `specweave init` 后工作。**

---

## 目的

通过智能分解和角色分配，协调多个专门代理以完成复杂的多步骤任务。

**架构**：层次化编排者-工作者模式
```
用户请求 → 编排者 → PM → 架构师 → 技术负责人 → 实施 → QA → 部署
```

---

## 何时激活

适用于需要 **3+ 代理** 或 **全面产品开发** 的请求：

| 用户说 | 需要的代理 | 模式 |
|--------|-------------|------|
| "为 X 构建一个 SaaS" | PM → 架构师 → 技术负责人 → 实施 → QA → DevOps | 顺序 |
| "创建实时聊天" | 架构师 → 后端 + 前端（并行） → QA | 并行 |
| "确保安全认证" | 安全 → 技术负责人 → 后端 → QA | 顺序 |
| "优化性能" | 技术负责人 → 性能 → 后端 → DevOps | 迭代 |

---

## 代理角色

### 战略层

**PM 代理 (pm-agent)**
- 产品战略、用户故事、优先级
- **何时**：启动新产品/功能

**架构师代理 (architect-agent)**
- 系统设计、ADR、技术栈
- **何时**：设计系统或主要功能

### 执行层

**技术负责人代理 (tech-lead-agent)**
- 技术规划、代码审查、质量标准
- **何时**：复杂技术决策

**后端代理**
- nodejs-backend, python-backend, dotnet-backend
- **何时**：服务器端实施

**前端代理 (frontend-agent)**
- React/Next.js，UI/UX 实施
- **何时**：构建用户界面

### 质量与运营层

**QA 负责人代理 (qa-lead-agent)**
- 测试战略、质量保证、覆盖率
- **何时**：定义测试方法

**安全代理 (security-agent)**
- 安全架构、威胁建模、合规性
- **何时**：安全关键功能

**DevOps 代理 (devops-agent)**
- 基础设施、CI/CD、部署、监控
- **何时**：运营任务

## 关键：安全编排模式

**规则**：编排者创建结构，引导用户在主要上下文中调用代理（而不是嵌套生成）。

### 阶段 0：首先创建增量结构

在调用任何代理之前，创建增量文件夹：

```typescript
// 1. 解析用户请求
const projectName = extractProjectName(userRequest);
// "event management" → "event-management"

// 2. 获取下一个编号
const nextNumber = getNextIncrementNumber();
// 例如，0001, 0002, 0003

// 3. 创建结构
const incrementPath = `.specweave/increments/${nextNumber}-${projectName}/`;
mkdir -p ${incrementPath}
mkdir -p ${incrementPath}logs/
mkdir -p ${incrementPath}scripts/
mkdir -p ${incrementPath}reports/

// 4. 创建占位文件
write ${incrementPath}spec.md (基本模板)
write ${incrementPath}plan.md (基本模板)
write ${incrementPath}tasks.md (基本模板)
write ${incrementPath}metadata.json (强制)
```

**spec.md 模板**：
```yaml
---
increment: 0001-project-name
title: "项目名称"
type: feature
priority: P1
status: planned
created: 2025-11-24
---

# 项目名称

## 概述
（由 PM 代理填写）

## 用户故事
（由 PM 代理填写）
```

**metadata.json 模板**：
```json
{
  "id": "0001-project-name",
  "status": "planned",
  "type": "feature",
  "priority": "P1",
  "created": "2025-11-24T12:00:00Z",
  "lastActivity": "2025-11-24T12:00:00Z"
}
```

### 阶段 1：引导用户通过代理工作流程

**向用户输出此工作流程**：

```
✅ 增量结构已创建： .specweave/increments/0001-project-name/

🎯 完整工作流程（在主要对话中运行这些命令）：

步骤 1：产品战略与需求
告诉 Claude: "完成增量 0001-project-name 的规格"
（PM 代理将自动激活）

步骤 2：架构与设计
告诉 Claude: "为增量 0001-project-name 设计架构"
（架构师代理将创建 ADR 和系统设计）

步骤 3：技术规划
告诉 Claude: "为增量 0001-project-name 创建技术计划"
（技术负责人代理将创建实施方法）

步骤 4：实施任务
告诉 Claude: "为增量 0001-project-name 创建任务"
（测试感知规划者将生成带测试的任务）

步骤 5：安全审查（如有需要）
告诉 Claude: "审查增量 0001-project-name 的安全性"
（安全代理将进行威胁建模）

步骤 6：实施
告诉 Claude: "实施增量 0001-project-name"
（后端/前端代理将实施代码）

步骤 7：质量保证
告诉 Claude: "运行增量 0001-project-name 的 QA"
（QA 代理将验证测试和覆盖率）

步骤 8：部署规划
告诉 Claude: "规划增量 0001-project-name 的部署"
（DevOps 代理将创建基础设施）

⚠️  在主要对话中顺序运行这些以防止上下文爆炸！
```

**不要使用 Task() 工具从此技能生成所有代理！**

## 编排模式

### 模式 1：顺序（默认）

**何时**：步骤之间存在依赖关系
```
PM → 架构师 → 技术负责人 → 后端 → 前端 → QA → DevOps
```

**用户工作流程**：
1. 创建增量结构
2. 引导用户逐个调用代理
3. 每个代理在下一个开始之前完成
4. 用户跟踪进度

### 模式 2：并行执行

**何时**：独立工作流
```
PM + 架构师（并行）
    ↓
后端 + 前端（并行）
    ↓
QA + DevOps（并行）
```

**用户工作流程**：
1. 创建增量结构
2. 确定并行机会
3. 引导用户：“您可以并行运行这些：[列表]”
4. 用户同时调用代理

### 模式 3：自适应（上下文感知）

**何时**：在执行过程中发现需求
```
PM → 架构师 → [发现需求] → 安全 → 技术负责人 → ...
```

**用户工作流程**：
1. 从基本计划开始
2. 代理发现额外需求
3. 在工作流程中间注入新代理
4. 动态调整计划

## 质量门（检查点）

### 门 1：在 PM 之后（需求完成）
**检查**：
- [ ] 用户故事定义了 AC-ID
- [ ] 成功标准可测量
- [ ] 依赖关系已识别
- [ ] 范围外定义

**决策**：继续架构或细化需求

### 门 2：在架构师之后（设计完成）
**检查**：
- [ ] 系统设计已记录
- [ ] ADR 已创建（≥3）
- [ ] 技术栈已选择
- [ ] 数据模型已定义

**决策**：继续实施或重新设计

### 门 3：在实施之后（代码完成）
**检查**：
- [ ] 所有 P1 任务已完成
- [ ] 测试通过（≥80% 覆盖率）
- [ ] 代码已审查
- [ ] 文档已更新

**决策**：继续部署或修复问题

### 门 4：在部署之前（生产就绪）
**检查**：
- [ ] 安全审查通过
- [ ] 性能可接受
- [ ] 监控已配置
- [ ] 回滚计划存在

**决策**：部署或解决阻碍

## 错误处理

### 代理失败
```
如果代理失败：
1. 捕获错误详情
2. 进行重试（最多 3 次尝试）
3. 如果可用，尝试后备代理
4. 如果是关键：暂停并通知用户
5. 永远不要静默失败
```

### 优雅降级
```
如果 PM 失败 → 使用基本模板
如果架构师失败 → 技术负责人创建简单设计
如果后端失败 → 创建手动实施指南
如果 DevOps 失败 → 创建手动部署步骤
```

### 用户干预
```
如果编排者卡住：
1. 显示当前状态
2. 向用户呈现选项
3. 让用户决定下一步
4. 从检查点恢复
```

## 最佳实践

**✅ 应该**：
- 首先创建增量结构
- 引导用户通过顺序工作流程
- 在阶段之间使用质量门
- 跟踪进度并向用户显示更新
- 优雅地处理错误并重试
- 允许用户跳过可选阶段

**❌ 不应该**：
- 一次性生成所有代理（上下文爆炸）
- 跳过增量创建
- 在没有质量门检查的情况下继续
- 在代理错误时静默失败
- 假设代理是无懈可击的
- 阻止用户手动干预

## 用户确认检查点

以下操作前**必须暂停并询问用户确认**：

| 检查点 | 触发条件 | 确认内容 |
|--------|---------|---------|
| 创建增量结构 | Phase 0 执行前 | 确认项目名称、类型和优先级，结构创建后不可轻易撤销 |
| 代理跳过或重排序 | 用户要求跳过某个质量门或阶段 | 确认跳过的风险和影响范围 |
| 热修复流程 | 检测到安全漏洞或紧急修复 | 确认是否走快速通道（4阶段）而非完整 8 阶段 |
| 架构师阶段技术栈选择 | 架构师输出 ADR 后 | 确认技术栈选择，此决策后续难以更改 |

## 错误处理与回退

| 错误场景 | 检测信号 | 回退策略 |
|---------|---------|---------|
| 代理执行失败 | 子代理返回错误或超时 | 重试最多 3 次；降级为手动实施指南输出给用户 |
| 质量门检查不通过 | 质量门 checklist 存在未通过项 | 暂停并展示缺失项，回退到上一个阶段修复 |
| 增量结构已存在 | 创建增量时目录已存在 | 提示用户选择：更新现有增量或创建新编号 |
| specweave 未初始化 | `.specweave/` 目录不存在 | 提示用户先运行 `specweave init`，或创建基础结构继续 |
