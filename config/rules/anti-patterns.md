# Anti-Patterns

跨领域的明确禁止项 —— 告诉 AI "不应该怎么做"。

---

## 测试 Anti-Patterns

| 禁止项 | 说明 | 正确做法 |
|--------|------|----------|
| 没有断言的测试 | 测试只运行代码，不验证结果 | 每个测试必须有 `expect`/`assert` |
| Mock 内部实现 | Mock 函数内部调用的其他函数 | 只 Mock 外部依赖（API、文件系统） |
| 测试覆盖无意义代码 | 测试 getter/setter、简单变量赋值 | 测试有业务逻辑的代码路径 |
| 硬编码测试数据 | 在测试中写死具体值作为"预期" | 使用有意义的边界值和等价类 |
| 测试之间共享状态 | 一个测试修改全局/模块状态影响其他测试 | 每个测试必须独立，可任意顺序执行 |

## 编码 Anti-Patterns

| 禁止项 | 说明 | 正确做法 |
|--------|------|----------|
| `var` 声明变量 | var 有作用域提升问题 | 使用 `const`（优先）或 `let` |
| Magic numbers | 代码中出现无解释的数字 | 定义常量并命名 |
| 深层嵌套 | if/else 超过 3 层 | 提取函数、使用卫语句、策略模式 |
| 函数过长 | 单个函数超过 50 行 | 拆分职责 |
| 重复代码 | 相同/相似代码出现 3 次以上 | 提取为共享函数 |
| 负面命名 | 使用否定形式命名（如 `isNotValid`） | 使用肯定形式（`isValid`） |

## 审查 Anti-Patterns

| 禁止项 | 说明 | 正确做法 |
|--------|------|----------|
| 跳过验证直接交付 | 未经独立审查直接给用户 | 必须经独立 Agent 校验 |
| 只修症状不修根因 | 哪里出错修哪里 | 找到根本原因，从源头修复 |
| 单次尝试定结论 | 修复一次就认为解决 | 验证问题确实不复现 |

## 设计 Anti-Patterns

| 禁止项 | 说明 | 正确做法 |
|--------|------|----------|
| 提前优化 | 为"未来可能"添加抽象 | YAGNI：先简单实现 |
| 过度工程 | 构建不必要的框架/基础设施 | 解决当前问题即可 |
| 缺乏边界处理 | 只考虑 happy path | 处理异常、空值、边界条件 |

## 安全 Anti-Patterns

| 禁止项 | 说明 | 正确做法 |
|--------|------|----------|
| 硬编码 secrets | API key、密码写在代码中 | 使用环境变量 |
| 未验证用户输入 | 直接使用用户传入的数据 | 始终验证类型、范围、格式 |
| SQL 字符串拼接 | `SELECT * FROM users WHERE id='${id}'` | 使用参数化查询 |
| 错误信息泄露敏感数据 | 抛出详细异常信息给用户 | 统一错误处理，隐藏细节 |
| XSS 漏洞 | 直接将用户输入插入 HTML | 使用转义库或框架自动转义 |
| 认证/授权缺失 | 未验证用户身份或权限 | 始终检查用户身份和权限 |

## 写作 Anti-Patterns

| 禁止项 | 说明 | 正确做法 |
|--------|------|----------|
| 事实无据 | 断言无具体来源（PubMed ID、DOI、数据库） | 每个事实断言必须有可验证的引用 |
| 链接伪造 | DOI/超链接无法解析或不存在 | 交付前验证所有链接有效 |
| 数据失真 | 统计数据与来源不一致 | 逐字核对来源数据 |
| 结论越界 | 结论超出证据支持范围 | 结论必须由证据推导 |
| 术语漂移 | 同一概念使用不同术语 | 全局保持术语一致性 |
| 跳过审查 | 未经独立 Agent 审查直接交付 | 必须经 quality-reviewer + verifier 校验 |
