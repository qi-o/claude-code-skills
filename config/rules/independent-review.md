# Independent Review Rules

## Core Principle: Producer ≠ Reviewer

**产出者 ≠ 审查者。** 任何复杂任务完成后，禁止由产出 Agent 自我审查后直接交付。
必须启动完全独立的 Agent 从零审视。

区别不在于谁更聪明——独立 Agent 用的是同一个模型。
区别在于它更「干净」：没有产出时的确认偏误，没有「这是我写的所以大概没问题」的惯性。

---

## 第一层：先调查后动手

**适用场景**：排查 bug、调查系统故障、分析异常行为。

| 阶段 | 行动 | 禁止 |
|------|------|------|
| 1. 调查 | 读日志、读源码、读配置 | ❌ 边看边改 |
| 2. 调研 | 查官方文档、GitHub Issues | ❌ 凭经验猜测就动手 |
| 3. 列计划 | 基于根因制定修复方案 | ❌ 跳过计划直接改文件 |
| 4. 校验 | 启动独立 Agent 并行验证计划 | ❌ 跳过校验直接执行 |
| 5. 呈现 | 把计划呈现给用户 | ❌ 不等确认就动手 |
| 6. 执行 | 按校验后的计划逐步执行 | ❌ 执行中临时加戏 |
| 7. 症状验证 | 复现原始触发场景，确认症状消失 | ❌ 只确认代码改了就收工 |

**关键**：调查和列计划可以是主 Agent 自己做，但第 4 步校验必须由独立 Agent 完成。第 7 步症状验证：单文件修复由执行者自验；涉及 3+ 文件的修复由独立 Agent 执行复现验证。

### 调试框架（在「调查」阶段使用）

**必须先找根本原因，再动手修复**

| 子阶段 | 行动 | 禁止 |
|--------|------|------|
| 1.1 读错误 | 仔细阅读错误信息和警告 | ❌ 跳过错误 |
| 1.2 复现 | 稳定触发问题，记录步骤 | ❌ 不可复现就猜 |
| 1.3 查变更 | 检查 git/依赖/配置变更 | ❌ 忽略环境差异 |
| 1.4 追踪数据流 | 找到源头，治本不治标 | ❌ 只修症状 |
| 1.5 模式分析 | 对比 working code 找差异 | ❌ 不查参考实现 |
| 1.6 写测试 | 修复前先写失败测试（复现问题） | ❌ 先改后测 |
| 1.7 假设测试 | 最小改动验证假设 | ❌ 一次改多个 |

**3次修复失败规则**：同一问题尝试 3 次修复后仍未解决，必须停下来质疑架构是否正确方向，而非继续修补。此时应：
- 回溯问题定义是否正确
- 考虑是否需要升级求助或用户介入
- 评估是否需要重新设计方案

## 第二层：计划校验

**适用场景**：任何涉及 3+ 文件修改、影响运行中服务、或用户明确要求的计划。

```
❌ 错误流程：主 Agent 制定计划 → 直接交付用户
✅ 正确流程：主 Agent 制定计划 → 独立 Agent 校验 → 修正 → 交付用户
```

独立 Agent 校验三个维度：
- **可行性**：文件/API/依赖是否实际存在
- **完整性**：是否遗漏边界情况、回滚方案
- **一致性**：步骤之间有无矛盾、前后依赖是否正确

禁止：跳过校验直接把初版计划交给用户。

## 第三层：Skill / 规则修改守门

**适用场景**：修改任何 SKILL.md、rules/*.md、CLAUDE.md 中的规则。

| 阶段 | 行动 | 禁止 |
|------|------|------|
| 1. 草拟修改 | 明确要改什么、改哪里 | ❌ 想到就直接写入 |
| 2. 冲突检测 | 启动独立 Agent，与现有规则逐条比对 | ❌ 自己检查自己的草案 |
| 3. 冲突处理 | 有冲突 → 呈现给用户裁决 | ❌ 发现冲突后自行裁决 |
| 4. 写入规则 | 按验证后的方案写入 | ❌ 偏离方案临时加戏 |
| 5. 覆盖率审计 | 独立 Agent 验证审查清单覆盖所有 BLOCKING 规则 | ❌ 跳过覆盖率验证 |
| 6. 回扫验证 | 独立 Agent 用新规则扫描触发本次修改的原始产出物 | ❌ 只写规则不验证效果 |

冲突检测检查三类问题：矛盾、语义重叠但措辞冲突、重复/可合并。

**BLOCKING 规则识别标准**：包含「禁止」「必须」「强制」「不得」关键词的规则条目视为 BLOCKING 规则，覆盖率审计必须逐条确认审查清单覆盖这些条目。

## 第四层：冰山协议

**适用场景**：任何任务执行中发现单点错误，且该错误可归因到规则/检查机制缺失。

```
❌ 错误做法：发现一个错误 → 直接修那个错误
✅ 正确做法：发现错误 → 归因到机制层面 → 修补机制 → 回扫同类产出
```

| 阶段 | 行动 | 禁止 |
|------|------|------|
| 1. 归因 | 判断：是哪条规则/检查缺失导致这个错误？ | ❌ 直接修单个错误 |
| 2. 修补机制 | 补充或修正对应规则/检查项 | ❌ 跳过机制修补 |
| 3. 回扫 | 独立 Agent 用修补后的机制扫描当前任务会话内的同类产出 | ❌ 只扫已知错误位置 |
| 4. 修复结果 | 按回扫结果修复所有同类问题 | ❌ 只修触发点 |

**触发条件**：错误可明确归因到「某条规则未定义」或「某项检查未覆盖」时触发。偶发性笔误等不触发。

**回扫范围**：当前任务会话内的同类产出，不要求扫描历史会话。

## 触发矩阵

| 任务类型 | 触发层级 | 独立 Agent |
|---------|---------|-----------|
| 系统故障排查 | 第一层（含调试框架） | `debugger` (sonnet) |
| 复杂修复计划（3+ 文件） | 第二层 | `verifier` (sonnet) |
| 架构/设计决策 | 设计工作流 | `architect` (opus) |
| 新功能/项目启动 | 设计工作流 | `planner` (opus) |
| Skill 规则修改 | 第三层 | `quality-reviewer` (sonnet) |
| 发现可归因错误 | 第四层（冰山协议） | `verifier` (sonnet) |
| 严肃写作产出 | 见 writing-quality.md | `quality-reviewer` + `verifier` |
| 单文件小改动 | 无需独立审查 | — |

---

## 分形原则

「产出者 ≠ 审查者」在每一个层级都适用：

- 产出物需要独立审查
- 审查的规则也需要独立审查
- 管理审查规则的流程，也需要独立审查

如果产出者和审查者是同一个系统，漏洞可能永远不会被发现——
因为那个系统会认为自己的检查是完整的。
